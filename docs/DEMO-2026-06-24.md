# Designbridge — Demo-Spickzettel (24.06.2026, 13:30)

## 1. Der 30-Sekunden-Pitch (zum Einstieg)

> „Designbridge ist eine **Brücke zwischen Design und Code**. Ich gebe ihm ein visuelles
> Design — einen Screenshot oder einen Figma-Frame — und es extrahiert daraus automatisch
> ein **echtes Design-System**: Farben, Typografie, Abstände, Radien, Schatten und ein
> Inventar der UI-Komponenten. Dieses System kann ich dann als **fertigen Code** exportieren —
> CSS-Variablen, Tailwind-Config oder das offene `tokens.json`-Format. Aus einem Bild wird
> also in Sekunden eine Grundlage, mit der Entwickler:innen sofort arbeiten können."

## 2. Wie es funktioniert (1 Satz, falls gefragt)

Der Screenshot geht an ein Backend, das **Claude (Sonnet) Vision** mit einem spezialisierten
Extraktions-Prompt aufruft. Das Modell liefert strukturierte Tokens + Komponenten-Inventar,
jeweils mit **Confidence** (high/med/low). Designbridge normalisiert das zu einem
quell-neutralen Modell und kann es über austauschbare „Emitter" als Code ausgeben.

## 3. Demo-Ablauf (genau diese Reihenfolge)

**Vorher:** Server läuft (`npm run dev`), Browser auf **http://localhost:5173**, Figma-PNG bereit.

1. **Leerer Zustand** — App startet mit Empty State → „New Import" klicken.
2. **Bild-Tab** — Figma-PNG reinziehen / auswählen → **„Analysieren"**.
   - *Sag währenddessen:* „Das geht jetzt an Claude Vision, dauert ein paar Sekunden."
3. **Erfolgs-Screen** — zeigt die gefundenen Kategorien mit Counts + Confidence.
   - *Talking Point:* „Es hat X Farben, Y Textstile, Komponenten erkannt — alles mit
     Vertrauenswert. Unsichere Werte sind markiert, nicht versteckt."
4. **„Open Library"** → durch die Library klicken:
   - **Dashboard** — Übersicht & Warnungen.
   - **Tokens** — die Farb-Swatches & Typo werden *visuell* gerendert. (Stärkster „Wow"-Moment.)
   - **Atomics/Components/Patterns** — als Karten (visuelle Nachbauten = Phase 3, ehrlich so benennen).
5. **Export-Tab** (der Abschluss) — links Format wählen:
   - **CSS-Variablen** → `:root { --color-… }`, unsichere mit Kommentar „bitte prüfen".
   - **Tailwind-Config** → fertige `theme.extend`-Struktur.
   - **tokens.json** → offenes W3C-Design-Token-Format.
   - *Talking Point:* „Das ist die Brücke zum Code — diese Datei droppt ein Dev direkt ins Projekt.
     **Kopieren** oder **Alle herunterladen**."

## 4. Was du NICHT live zeigst (als Roadmap framen)

## 5. Vision in einem Satz (zum Abschluss)

> „Das Ziel ist eine **bidirektionale** Brücke: ein quell-neutrales Design-System in der Mitte,
> das man aus Figma, Screenshots, URLs oder Code *befüllen* und in beide Richtungen *ausgeben*
> kann. Was ihr heute seht, ist die erste vollständige Strecke: vom Bild zum Code."

## 6. Sicherheitsnetz (eingebaut & aktiv)

Es ist ein **automatischer Fallback** scharf geschaltet (`DEMO_FALLBACK=1` in `.env`): Wenn der
Live-Scan fehlschlägt (z. B. API-Credits leer), liefert der Server statt eines Fehlers einen
**hinterlegten Datensatz, der zu genau diesem CRAFTUI-Dashboard passt** (`server/fixtures/demo-dashboard.json`).
Die Demo läuft dadurch **optisch komplett live** (Upload → Fortschritt → Ergebnis → Export) —
nur die Zahlen kommen aus dem Fixture statt aus dem Modell.

**Zwei Zustände:**
- **Credits aufgeladen** → echter Scan analysiert das hochgeladene Bild. Fallback bleibt stumm.
- **Credits noch leer** → jedes hochgeladene Bild liefert denselben Dashboard-Datensatz.
  → In dem Fall in der Demo **genau dieses Dashboard (`Testdaten/Reports/02.png`) hochladen**,
  damit Bild und Ergebnis zusammenpassen.

**Nach der Demo:** `DEMO_FALLBACK=1` in `.env` auf `0` setzen / entfernen → normales Verhalten.
